# Option Value Vocabulary & Resolve-Map Pattern

Intensity-dial options use a unified `'low' | 'high'` vocabulary with fused resolve+map so that consumers — from simple enum callers to LaunchDarkly integrations — always flow through normalization.

## Context

Over 5 rounds of config-surface widening, verblets accumulated ~47 resolvable options. The named-policy options used inconsistent vocabularies — ~30 different words for what are mostly intensity spectrums (`'strict'|'moderate'|'loose'`, `'guided'|'exploratory'|'challenging'`, `'focused'|'balanced'|'high'`). A consumer had to know that `compressionRatio: 0.15` means aggressive compression, that `llmWeight: 0.9` means LLM-heavy ranking, and that `diversityLevel: 'focused'` and `dialogueStyle: 'guided'` both mean "less of the thing." Each option had its own vocabulary, its own scale direction, and its own domain-specific numbers.

Separately, consumers using dynamic configuration systems (LaunchDarkly, remote config) need to supply values that are resolved lazily with full nested context — the composed operation path, logger, and domain attributes. An LD flag returning `'high'` at resolution time needs to flow through the same normalization as a static `compression: 'high'` passed at call time. The mapping from consumer-visible values to chain internals needs to be a first-class part of the resolve pipeline, not bolted on after.

## Decisions

| Decision | Reason | Rejected Alternative |
|---|---|---|
| Two named levels: `'low'` and `'high'` | Most behavioral options are intensity dials — "more or less of what the option name says." The option name carries the semantics: `compression: 'high'` = aggressive, `preservation: 'high'` = strict. No need to remember domain words. `undefined` means default behavior — you only speak up to say "I want more" or "I want less." | Per-option vocabularies like `'strict'|'moderate'|'loose'` (30 different word sets to memorize, no consistency across chains), three-level `'low'|'medium'|'high'` (medium is always the default — naming the thing you get by not passing the option adds noise without information) |
| Exported pure mapper functions per chain | Each chain exports a `mapX` that normalizes any input (enum string, raw primitive, `undefined`) to the concrete value the chain needs. Consumers can import these for validation, display, or wrapping with external systems. Raw primitive passthrough means power consumers aren't capped at two levels. | Centralized mapping registry (couples all chains to a shared module, harder to tree-shake), no export (consumers using LD or validation can't access the normalization logic) |
| Fused resolve+map via `resolveMapped` / `resolveOptionMapped` | The mapper runs inside the resolve call, not after it. Chains call `resolveMapped('compression', options, mapCompression)` — one call, always normalized. This guarantees that whatever comes through the option hierarchy (static value, `policy` function) gets normalized before the chain sees it. Chains can't forget the mapping. | Resolve then map as two separate steps (every chain must remember to apply the mapper — one missed call means raw LD strings reaching code that expects numbers), mapper as middleware in the resolve pipeline (over-engineering for what is a one-line function composition) |
| Three mapper shapes: numeric, structured, validating | Numeric mappers (enum → number, number passthrough) cover most cases. Structured mappers (enum → object, object passthrough) handle multi-field configs like `preservation: { short, long }`. Validating mappers (enum → enum, undefined passthrough) handle options where the chain uses the string directly. All three follow the same contract: accept `undefined`, enum, or raw value; return the typed value the chain needs. | Single mapper shape (forces awkward encoding of structured options into numbers), no mapper for validating options (inconsistent — some options go through `resolveMapped`, others don't) |
| Domain-specific values for qualitative modes | Options that represent genuinely different strategies (not a spectrum) keep their own words: `thresholdStrategy: 'statistical'|'percentile'|'fixed'`, `accumulatorMode: 'auto'|'collection'|'scalar'`, `progressMode: 'none'|'coarse'|'batch'|'detailed'`. These aren't "more or less" — they're different algorithms or structural shapes. | Force everything into `'low'|'high'` (loses meaning — `thresholdStrategy: 'high'` says nothing), separate naming convention for non-intensity options (unnecessary distinction when the existing names already communicate clearly) |
| Lazy evaluation with nested operation context | `policy` functions receive full context at resolution time — including composed operation paths from `withOperation` (e.g. `'document-shrink/score'`). An LD flag can return different values for the scoring sub-operation than for the parent chain. The mapper is applied automatically after resolution, so LD consumers just return raw flag values. | Eager evaluation at call time (loses nested context — can't differentiate sub-operations), consumer-side mapping (every LD integration must import and call the mapper manually) |
| Structured mappers as operating modes | Some options coordinate multiple internal decisions rather than tuning a single value. `thoroughness` in document-shrink gates three LLM phases and shifts token allocation with one option. The mapper returns a decision object — a compact structured token bundling related decisions for coherency. This implements the permanent-flags "operating mode" pattern: one change yields a predictable, consistent set of effects. | Individual gate options per phase (more composable but requires the consumer to understand internal phases — coordination mistakes are easy), single boolean (too coarse — "fast" and "thorough" differ in more than on/off) |
| Classification posture as a prompt-shaping option | `sensitivity` in calibrate steers the LLM's false-positive/false-negative tradeoff through prompt guidance rather than post-hoc filtering. Low sensitivity instructs conservative classification; high sensitivity instructs aggressive detection. The option shapes the specification that governs all downstream classifications — one decision with consistent, scaled effect. | Post-classification filtering (loses the LLM's judgment about borderline cases — you can't un-miss something the spec didn't look for), numeric threshold adjustment (calibrate's value comes from LLM-calibrated criteria, not numeric cutoffs) |
| Anchor strategy as a scoring quality dial | `anchoring` in score controls how the first batch's results calibrate subsequent batches. No anchors means batches are independent (faster, less consistent). Rich anchors include extremes plus median items (slower first batch, tighter cross-batch calibration). The recurring tradeoff: anchor quality vs. scoring consistency across batches. | Fixed anchor strategy (hardcoded min/max extraction works for most cases but isn't optimal for skewed distributions or small corpora), consumer-side anchor injection (adds complexity to every score consumer) |
| Detection intensity as a threshold dial | `detection` in probe-scan maps `'low'|'high'` to cosine similarity thresholds. Low detection raises the threshold (0.55) — fewer hits, fewer false positives. High detection lowers it (0.3) — more hits, catches weaker signals. The inverted scale matches the option semantics: "more detection" means catching more. Raw number passthrough for power consumers who need precise control. | Keeping the raw `threshold` option (requires knowing that lower numbers = more sensitive — counterintuitive), separate `sensitivity` option (name collision with calibrate's `sensitivity` which shapes prompts, not thresholds) |
| Category granularity as prompt guidance | `granularity` in group steers the category discovery phase via prompt guidance. Low granularity instructs "fewer, broader categories — merge aggressively." High granularity instructs "finer-grained categories — preserve distinctions." The guidance is injected as a tagged XML block, shaping the LLM's merge/split decisions during the reduce pass. | Numeric `maxCategories` cap (constrains the output but doesn't guide the LLM's grouping strategy — you get N categories but they may be poorly chosen), post-hoc category merging (loses the LLM's judgment about which distinctions matter) |
| Filter strictness as borderline handling | `strictness` in filter controls the LLM's decision on ambiguous items via prompt guidance. Low strictness instructs "when uncertain, include" (fewer false negatives). High strictness instructs "when uncertain, exclude" (fewer false positives). This shapes every batch's filtering prompt — one decision that consistently biases the yes/no boundary. | Numeric confidence threshold (filter returns yes/no decisions, not scores — there's no numeric value to threshold), post-filter pass to review borderline items (doubles LLM cost for edge cases) |
| Advice depth as an introspection gate | `advice` in expect controls whether failed assertions trigger module introspection — an LLM call to identify the module under test plus file reads of its source. Low advice skips introspection entirely (faster, cheaper, no extra LLM call). High advice (and the default) does full introspection. This is the gate effect shape: one option enables or disables an expensive diagnostic step. | Always-on introspection (wastes an LLM call when consumers just want pass/fail without debugging context), separate `introspection` boolean (less discoverable than the unified `'low'|'high'` vocabulary) |
| Sorting effort as an operating mode | `effort` in sort coordinates three interacting parameters — `iterations`, `extremeK`, `selectBottom` — into a single posture. Low effort does one pass with a small extremes window and only finds the top (fast, rough ordering). High effort does multiple passes with a larger window and finds both ends (expensive, precise full ordering). The mapper returns a decision object; explicit parameter overrides still take precedence via the resolve fallback chain. | Exposing three separate parameters (requires understanding the pairwise-extremes algorithm — consumers want "sort roughly" vs. "sort carefully," not `iterations: 2, extremeK: 15`), single `iterations` parameter (doesn't capture the extremeK/selectBottom coordination) |
| Truncation aggression as a threshold dial | `aggression` in truncate maps `'low'|'high'` to score thresholds for content removal. Low aggression (threshold 7) only removes content the LLM strongly considers removable — conservative. High aggression (threshold 4) removes anything the LLM isn't confident should stay — aggressive. Replaces the raw `threshold` option whose direction was unintuitive (lower number = more aggressive). | Keeping raw `threshold` (consumers must know that lower numbers remove more — backwards from expectation), separate `conservative` boolean (too coarse for a continuous tradeoff) |
| Exploration breadth as a search strategy dial | `exploration` in questions maps `'low'|'high'` to search breadth controlling the depth/breadth tradeoff in iterative question generation. Low exploration (0.3) narrows the subset of generated questions kept each round and triggers depth-first drilling into random questions — focused, niche angles. High exploration (0.8) keeps most generated questions — maximum breadth and diversity. Replaces the raw `searchBreadth` float whose meaning was opaque. | Keeping raw `searchBreadth` (a float between 0 and 1 with no semantic anchor — consumers can't predict what 0.4 vs. 0.6 means), separate depth/breadth boolean (the chain's behavior is a continuum, not a binary switch) |
| Timeline enrichment as an operating mode | `enrichment` in timeline coordinates three pipeline phases — LLM deduplication, knowledge base building, and enrichment mapping — into a single posture. Low enrichment skips all three (extraction + deterministic merge only — cheapest). Default enables LLM dedup only. High enrichment activates all phases including knowledge base construction and enrichment mapping. The mapper returns a decision object (`{ llmDedup, knowledgeBase, enrichMap }`). | Separate `enrichWithKnowledge` boolean (only gates one phase, can't independently control LLM dedup), three separate booleans (consumers must understand internal pipeline phases to configure correctly) |
| Date rigor as a validation gate | `rigor` in date controls the expectation-generation and validation loop. Low rigor skips expectations and validation entirely — extraction only (1 LLM call, cheapest). Default extracts + validates + retries, returning best-effort date on exhaustion. High rigor uses more validation attempts and returns `undefined` on exhaustion instead of a best-effort date. The mapper returns a decision object (`{ validate, maxAttempts, returnBestEffort }`). | Keeping validation always-on (wastes 2+ LLM calls when consumers just need a quick extraction), separate `skipValidation` boolean (doesn't capture the maxAttempts/returnBestEffort coordination) |
| Veiled variant coverage as an operating mode | `coverage` in veiled-variants coordinates strategy count and variants per strategy. Low coverage runs 1 strategy with 3 variants (1 LLM call, 3 results). Default runs all 3 strategies with 5 variants each (3 calls, 15 results). High coverage runs all 3 strategies with 8 variants each (3 calls, 24 results). Explicit `strategies` or `variantCount` still override via the resolve fallback chain. | Exposing `strategies` and `variantCount` separately (consumers must understand the strategy names and pick variant counts without guidance), single `variantCount` parameter (doesn't capture the strategy subset coordination) |
| Join fidelity as an operating mode | `fidelity` in join coordinates window granularity and overlap density. Low fidelity uses large windows (10) with minimal overlap (25%) — fewer LLM calls, rougher transitions. Default uses moderate windows (5) with 50% overlap. High fidelity uses small windows (3) with high overlap (67%) — more LLM calls, smoother transitions. Explicit `windowSize` or `overlapPercent` still override. | Exposing `windowSize` and `overlapPercent` separately (consumers must understand the windowed stitching algorithm to set coherent values), single `windowSize` parameter (doesn't capture that overlap density should scale with window granularity) |
| Pattern detection thoroughness as an operating mode | `thoroughness` in detect-patterns coordinates accumulator capacity and output limit. Low thoroughness uses a small accumulator (20 items, topN 3) — fast scan for dominant patterns, rare patterns evicted early. Default uses capacity 50 and topN 5. High thoroughness uses a large accumulator (100 items, topN 10) — rare patterns survive longer, deeper analysis. Capacity is embedded in the reduce prompt, so it directly affects token usage per batch. Explicit `capacity` or `topN` still override. | Exposing `capacity` and `topN` separately (consumers must understand the accumulator eviction algorithm to set coherent values), single `capacity` parameter (doesn't capture that output limit should scale with accumulator size) |
| Block detection precision as an operating mode | `precision` in extract-blocks coordinates window size and overlap for boundary detection. Low precision uses large windows (200 lines) with minimal overlap (10) — ~5 calls for 1000 lines, may miss subtle boundaries. Default uses windows of 100 with overlap 20. High precision uses small windows (50) with large overlap (30) — ~50 calls for 1000 lines, catches boundaries near window edges. 10x cost difference between low and high. Explicit `windowSize` or `overlapSize` still override. | Exposing `windowSize` and `overlapSize` separately (consumers must understand the rolling-window algorithm to pick coherent values), single `windowSize` parameter (doesn't capture that overlap should scale with window granularity for boundary coverage) |
| Query variant divergence as prompt guidance | `divergence` in embed-multi-query injects guidance controlling variant diversity. Low divergence instructs tight paraphrases with similar phrasing. High divergence instructs maximally diverse variants with contrasting terminology. Verblet prompt-shaping options return a guidance string (or `undefined`) rather than a decision object — no `resolveMapped`, no `withOperation`. | Numeric `diversityScore` float (opaque — consumers can't predict behavior), exposing strategy parameters separately (embed-multi-query is a single-call verblet, not an orchestrating chain) |
| Fill creativity as prompt guidance | `creativity` in fill-missing injects guidance controlling the confidence/risk tradeoff for imputation. Low creativity instructs conservative fills with `[UNKNOWN]` for uncertain values. High creativity instructs speculative fills attempting every gap. Same verblet prompt-shaping pattern as `divergence`. | Numeric `confidenceThreshold` (doesn't capture the behavioral shift between conservative and speculative — it's not just a threshold, it's a whole approach), always-conservative mode (useful for pipelines but loses the human-review use case) |
| Commonality analysis depth as prompt guidance | `depth` in commonalities controls whether analysis finds surface-level literal features or deep abstract/structural connections. Low depth instructs "directly observable shared features, no inferred abstract connections." High depth instructs "structural patterns, functional relationships, non-obvious conceptual links." The option fundamentally changes the nature of insights returned — feature comparisons need `low`, thematic analyses need `high`. | No option (hardcoded "common elements, shared features" — always surface-level), separate `abstractionLevel` numeric parameter (the spectrum isn't numeric — it's about the character of analysis) |
| Intent matching tolerance as prompt guidance | `tolerance` in intent controls how strictly the operation matching and parameter extraction behaves. Low tolerance instructs "only match if highly confident, set operation to null if ambiguous, only extract explicitly stated parameters." High tolerance instructs "always select closest match, infer reasonable parameter values from context." A recurring tradeoff: chatbot routing wants lenient matching; API validation wants strict matching. | Separate `confidenceThreshold` float (intent returns operation names, not scores — there's no numeric value to threshold), always-best-match behavior (safe for chatbots but dangerous for API dispatch) |
| Step-back abstraction level as prompt guidance | `abstraction` in embed-step-back controls how far the generated questions generalize from the original query. Low abstraction instructs "one level more general, same domain and vocabulary." High abstraction instructs "foundational principles, first principles, cross-domain analogies." Directly shapes RAG retrieval quality: customer support needs close step-backs (nearby articles), research assistants need foundational ones (theoretical papers). | Fixed "step back to concepts and principles" (no control over distance — always moderate), numeric `abstractionLevel` (the distance isn't numeric — it's about conceptual layers) |
| Query decomposition granularity as prompt guidance | `granularity` in embed-subquestions controls decomposition depth. Low granularity instructs "2-3 broad sub-questions covering major facets." High granularity instructs "many fine-grained sub-questions each targeting a single fact." Controls the retrieval strategy: coarse decomposition yields fewer broader retrievals for survey answers; fine decomposition yields many targeted retrievals for comprehensive fact-checking. | Fixed "atomic sub-questions" (undefined atomicity — LLM decides), numeric `maxQuestions` cap (constrains quantity but doesn't guide the decomposition strategy) |

## Configuration tiers

Three consumer tiers, each building on the resolve-map pipeline:

**Enum consumers** pass `'low'` or `'high'`. The mapper converts to the concrete value. This is the common case — no domain knowledge required.

**Power consumers** pass raw primitives (numbers or objects). The mapper passes them through unchanged. This covers the gap when two levels aren't precise enough.

**Dynamic consumers** supply `policy` functions that are evaluated lazily at resolution time with full nested context. The mapper is applied automatically — the consumer returns whatever the flag service gives them. This is the LD/remote-config integration path.

## Operating modes and strategic options

Some options go beyond intensity dials — they coordinate multiple internal decisions into a single posture. These implement the permanent-flags concept of operating modes: "prevents partial or mixed states, makes response faster, makes review easier."

**`thoroughness`** (document-shrink) — coordinates all internal phases of the most complex chain. Low thoroughness skips all three LLM calls (query expansion, edge scoring, compression) and relies on pure TF-IDF selection — fast, cheap, no LLM cost. High thoroughness activates all phases and shifts the token budget to favor more compression — thorough, expensive, maximum quality. The mapper returns a decision object that the chain reads at each phase gate.

**`anchoring`** (score) — controls the calibration strategy for batched scoring. Score processes items in batches; the first batch establishes "anchor" examples (scored items that subsequent batches reference for consistency). Low anchoring skips anchors entirely — batches score independently. High anchoring builds richer anchors with extremes plus median items from the first batch, tightening calibration across all subsequent batches.

**`sensitivity`** (calibrate) — steers the classification posture of the calibration specification. Low sensitivity produces a conservative spec that prefers false negatives — only strong signals get flagged. High sensitivity produces an aggressive spec that prefers false positives — weak or ambiguous signals are caught. This shapes the specification itself, so the posture consistently affects every item classified against it.

**`detection`** (probe-scan) — maps to a cosine similarity threshold for embedding-based scanning. Low detection (0.55) requires stronger matches, reducing false positives. High detection (0.3) catches weaker signals, reducing false negatives. The inverted numeric scale matches the option semantics: "more detection" finds more.

**`granularity`** (group) — steers the category discovery phase via prompt guidance. Low granularity produces fewer, broader categories by instructing aggressive merging. High granularity preserves finer distinctions between items. The guidance shapes the reduce pass that builds the taxonomy — one prompt-shaping decision that flows through to all subsequent assignment batches.

**`strictness`** (filter) — controls the LLM's handling of borderline items via prompt guidance. Low strictness instructs inclusion when uncertain (fewer false negatives). High strictness instructs exclusion when uncertain (fewer false positives). Injected into every batch's filtering prompt as a tagged guidance block.

**`advice`** (expect) — gates the module introspection step in failed assertion advice. Low advice skips the LLM call to identify the module under test and the file reads of its source — faster and cheaper. High advice (the default) does full introspection for richer debugging context. This is the gate effect shape from the permanent-flags theory.

**`effort`** (sort) — coordinates the pairwise-extremes sorting algorithm's three interacting parameters. Low effort does a single pass with a small extremes window (`extremeK: 5`) and only extracts the top — fast, rough ordering suitable for top-K selection. High effort does two passes with a larger window (`extremeK: 15`) and extracts both ends — expensive but produces a much more precise full ordering. The mapper returns a decision object; explicit parameter overrides (`iterations`, `extremeK`, `selectBottom`) still take precedence through the resolve fallback chain.

**`aggression`** (truncate) — controls how aggressively trailing content is removed. Low aggression uses a higher score threshold (7) — only removes content the LLM is very confident should go. High aggression uses a lower threshold (4) — removes anything the LLM isn't strongly convinced should stay. Replaces the raw `threshold` option whose numeric direction was counterintuitive.

**`exploration`** (questions) — controls the depth vs. breadth tradeoff in iterative question generation. Low exploration (0.3) narrows the kept subset each round and triggers depth-first drilling into random questions — focused, niche discovery. High exploration (0.8) keeps most generated questions and lets the LLM pick the most interesting drill-down target — maximum diversity. Replaces the raw `searchBreadth` float.

**`enrichment`** (timeline) — coordinates the three post-extraction phases of timeline processing. Low enrichment skips all three phases — extraction plus deterministic deduplication only, no LLM calls beyond chunk extraction. Default enables LLM-based deduplication but skips knowledge enrichment. High enrichment activates the full pipeline: LLM dedup, knowledge base construction via reduce, and enrichment mapping via map. The mapper returns a decision object; object passthrough lets power consumers enable any subset of phases independently.

**`rigor`** (date) — gates the expectation-generation and validation loop in date extraction. Low rigor skips the expectations LLM call and the bool-validation loop entirely — a single extraction call, returning whatever the LLM gives. Default runs the full pipeline with 3 validation attempts, returning the best-effort date on exhaustion. High rigor increases validation attempts to 5 and returns `undefined` on exhaustion instead of a potentially wrong date. The mapper coordinates `validate`, `maxAttempts`, and `returnBestEffort` into one posture.

**`coverage`** (veiled-variants) — coordinates how much of the veiling space to explore. Low coverage runs a single strategy (scientific framing) with 3 variants — a quick probe yielding 3 results from 1 LLM call. Default runs all 3 strategies with 5 variants each — 15 results from 3 parallel calls. High coverage runs all 3 strategies with 8 variants each — 24 results from 3 calls, maximum diversity for comprehensive filter evasion. Explicit `strategies` or `variantCount` still override via the resolve fallback chain, so power consumers can mix coverage with specific strategy selection.

**`fidelity`** (join) — coordinates the windowed stitching algorithm's granularity and overlap density. Low fidelity processes 10 fragments per window with 25% overlap — fewer, coarser passes for quick joins where transition quality is secondary. Default processes 5 per window with 50% overlap — balanced quality/cost. High fidelity processes 3 per window with 67% overlap — maximum context per fragment, more stitch calls, smoother transitions for publishable output. The mapper returns `{ windowSize, overlapPercent }`; explicit parameter overrides still take precedence.

**`thoroughness`** (detect-patterns) — coordinates the accumulator's capacity and output limit for pattern detection via reduce. Low thoroughness uses a 20-item accumulator with topN 3 — dominant patterns only, rare patterns evicted early, smaller accumulator JSON per batch. Default uses capacity 50 with topN 5. High thoroughness uses a 100-item accumulator with topN 10 — rare and subtle patterns survive the eviction pressure across many batches. The capacity value is embedded directly in the reduce prompt instructions, so it shapes the LLM's eviction strategy as well as token usage.

**`precision`** (extract-blocks) — coordinates window size and overlap for block boundary detection in large documents. Low precision uses 200-line windows with 10-line overlap — fast scan for obvious boundaries, ~5 LLM calls per 1000 lines. Default uses 100-line windows with 20-line overlap — ~12 calls. High precision uses 50-line windows with 30-line overlap — thorough detection catching boundaries near window edges, ~50 calls per 1000 lines. The 10x cost difference makes this a high-leverage option for large document processing.

**`divergence`** (embed-multi-query verblet) — prompt-shaping option controlling query variant diversity. Low divergence injects guidance to stay close to the original query with minor keyword variations — tight paraphrases that improve retrieval precision. High divergence instructs maximally diverse variants using contrasting terminology and tangential concepts — each variant should retrieve documents the others would miss. The guidance string is injected into the multi-query prompt template. Unlike chain operating modes, verblet prompt-shaping options return a guidance string (or `undefined`) rather than a decision object, preserving the single-call verblet's simplicity.

**`creativity`** (fill-missing verblet) — prompt-shaping option controlling the confidence/risk tradeoff when imputing missing values. Low creativity instructs conservative fills — only high-confidence candidates, `[UNKNOWN]` for anything uncertain, confidence scores below 0.3 for unsupported fills. High creativity instructs speculative fills — best educated guess for every gap, plausible candidates preferred over `[UNKNOWN]`, honest confidence scores but always attempting a fill. The guidance string is appended to the fill-missing prompt.

**`depth`** (commonalities verblet) — prompt-shaping option controlling analysis depth for finding shared features across items. Low depth instructs surface-level analysis — literal, directly observable shared features such as physical properties, obvious category membership, and explicit shared attributes. High depth instructs deep structural analysis — underlying mechanisms, analogous roles, philosophical parallels, and non-obvious conceptual links. The option fundamentally changes the *character* of the output: a feature comparison matrix versus a thematic insight report. This is a non-cost strategic option — the LLM call is identical, only the prompt guidance differs.

**`tolerance`** (intent verblet) — prompt-shaping option controlling intent matching strictness and parameter extraction confidence. Low tolerance instructs strict matching — only select an operation if highly confident, set operation to null if ambiguous, only extract explicitly stated parameters. High tolerance instructs lenient matching — always select closest match, infer reasonable parameter values from context, prefer dispatching over returning no match. A recurring deployment-context tradeoff: chatbot routing benefits from lenient matching (always route somewhere), while API validation benefits from strict matching (reject ambiguous requests).

**`abstraction`** (embed-step-back verblet) — prompt-shaping option controlling how far step-back questions generalize from the original query. Low abstraction generates questions one level more general — broader phrasing of the same topic, same domain and vocabulary. High abstraction generates questions about foundational principles, first principles, and cross-domain analogies. Directly shapes what kind of background context RAG retrieval finds: practical how-tos (low) versus theoretical foundations (high).

**`granularity`** (embed-subquestions verblet) — prompt-shaping option controlling query decomposition depth. Low granularity decomposes into 2-3 broad sub-questions covering major facets. High granularity decomposes into many fine-grained sub-questions each targeting a single specific fact, entity, or data point. Controls the downstream retrieval strategy: coarse decomposition yields fewer, broader retrievals for survey-style answers; fine decomposition yields many targeted retrievals for comprehensive fact-checking or detailed reports.

## Constraints

- Mappers are pure functions with no side effects — they can be called outside chains for validation, display, or pre-computation
- `resolveMapped` is the only way chains consume mapped options — no raw `resolve` + manual mapping
- `undefined` always means "use the chain's default behavior" — it is never an error
- Unknown enum strings (typos, old names) fall through to the default, same as `undefined` — no throws
- Old option names (`diversityLevel`, `dialogueStyle`, `compressionRatio`, `llmWeight`) are breaking changes — consumers passing them will silently get default behavior
- Raw passthrough means the mapper never restricts what a power consumer can do — `compression: 0.2` works even though it's between `'low'` (0.45) and `'high'` (0.15)
